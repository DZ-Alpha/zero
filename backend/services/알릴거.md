# 프론트 연동 전 확인 요청 — API 명세서 대비 변경/차이점

> 작성일: 2026-07-16 (2026-07-18 갱신 — `기능명세서-20260718-2144.xlsx` 반영, 5번 항목 추가)
> 대상 문서: `기능명세서.xlsx` (진짜기능명세 / 진짜API-Spec / 제로파인더 기능명세 시트)
> 대상 서비스: login, admin, main, product, ingredients, diet, community, recipe (총 8개)

프론트 연동 시작 전에 팀장님/프론트에 꼭 공유해야 할 사항을 정리했습니다. 백엔드 코드와 최신 명세서를 실제로 대조해서 나온 **진짜 차이점**만 담았고, 이미 구두로 합의된 사항(중복 구현 허용 등)은 제외했습니다.

---

## 0. `{DOMAIN}/b/{경로}` 프리픽스 — 확인 완료

API-Spec 시트의 "Backend Path"는 `{DOMAIN}/b/{경로}` 형태로 붙는데, **게이트웨이에서 `/b`를 붙여주는 구조로 이미 잡혀 있습니다.** 프론트 담당이 `.env`로 유연하게 설정할 예정이라, 각 서비스 코드(`/home/search`, `/admin` 등 `/b` 없는 경로 그대로)는 수정할 필요 없습니다. 별도 조치 불필요.

---

## 1. 명세서 자체의 정합성 문제 (RC-01xx 번호 꼬임)

`진짜기능명세` 시트와 `진짜API-Spec` 시트에서 RC-0104 이후 번호가 서로 어긋나 있습니다.

| ID | 기능명세 시트 | API-Spec 시트 |
| --- | --- | --- |
| RC-0104 | 제품 사진 분석 (OCR) | `/diet/other-foods` (저장된 분석 결과 조회) |
| RC-0105 | 다른 제품 추천 (대체 상품) | `/diet/calender` (캘린더) |
| RC-0106 | 캘린더 | `/receipe/low-sugar` (저당 레시피 목록) |

API-Spec 시트에는 RC-0103(`/diet/ai-analyze`)과 RC-0104(`/diet/other-foods`) 사이에 **ID가 없는 새 행** `/diet/ocr-analyze`가 끼어 있는데, 이게 기능명세 시트의 RC-0104(OCR 분석)에 해당하는 것으로 보입니다. 즉 API-Spec 시트에 새 행을 추가하면서 뒤 번호를 안 밀어서 두 시트가 어긋난 상태입니다.

저희 diet-service 코드 내부 주석(RC-ID 라벨)은 **예전 번호 체계**(RC-0105=대체상품, RC-0106=캘린더)로 달려 있어서 지금 API-Spec 시트 번호와도 안 맞습니다. **실제 URL 경로 자체는 문제없이 동작**하니 급한 버그는 아니지만, 명세서 ID 정리가 먼저 되어야 서로 헷갈리지 않을 것 같습니다.

## 3. 구현은 됐는데 명세서엔 아직 없는 것 (경로 재확인 필요)

- **레시피 대체상품 추천** (기능명세 RC-0109, "기존 레시피 재료에서 대체할 수 있는 제로/저당 제품 추천")은 실제로 구현돼 있습니다(`GET /recipes/{id}/substitutes`). 다만 API-Spec 시트엔 이 기능의 URL 행 자체가 없어서, 프론트가 참조할 공식 계약이 없는 상태입니다 → 시트에 행 추가 요청 필요.
- recipe-service 경로는 `/recipes/*` (복수, 정상 철자)인데, 명세서엔 `/receipe/*` (오타, 단수)로 적혀 있습니다. 오타를 명세서에서 고칠지, 저희가 오타 경로로 맞출지 결정이 필요합니다 (권장: 명세서 오타 수정).
- main-service에 `/home/health-profile`(GET/PUT), `/home/preferences`(GET/POST/DELETE), `/home/me`(GET)가 있는데, 이건 API-Spec 시트엔 없는 엔드포인트입니다. 이전에 "로그인 유저 맞춤 정보 제공"을 메인 서비스에 두기로 직접 지시하셨던 부분이라 의도된 추가이지만, 명세서에도 반영해두는 게 좋을 것 같습니다.

## 4. 참고 — 의도된 중복 (이미 구두 확인됨, 재론 불필요)

- `GET /search`: main-service(PREPARING placeholder)와 product-service(실제 구현) 둘 다 보유. **게이트웨이 라우팅 시 실제 트래픽은 product-service로 가야 합니다** — main-service 쪽으로 가면 빈 결과만 나옵니다.
- 인기 랭킹(`/home/rank/item`, MN-0110)과 감미료 정보(main-service 계열 vs ingredients-service `/community/gam-list`)도 기존에 안내드린 대로 중복 허용된 부분이라 그대로 유지합니다.

## 5. 2026-07-18 명세서 갱신분 — 신규 구현 완료

`기능명세서-20260718-2144.xlsx`로 명세서가 갱신되면서 아래 10개 항목(MN-0114, PR-0307~0308, RC-0111~0117)이 새로 추가됐습니다. MN-0114(로딩/빈결과/오류 등 공통 상태 UI)는 프론트 전용이라 백엔드 조치가 없고, 나머지 9개는 이번에 전부 구현해서 반영했습니다.

| 기능ID | 경로 | 설명 |
| --- | --- | --- |
| PR-0307 | `POST /product/favorite` | 상품 찜 등록/해제 토글 |
| PR-0308 | `GET /product/favorite/list` | 찜한 상품 목록 |
| RC-0111 | `POST /recipes/favorite` | 레시피 찜 등록/해제 토글 |
| RC-0112 | `GET /recipes/favorite/list` | 찜한 레시피 목록 (응답 키는 명세 원문 그대로 `list-receipe`) |
| RC-0113 | `POST /diet/records` | 식단 기록 생성 (레시피/상품/사진 공통 모델) |
| RC-0114 | `PUT /diet/records/{id}` | 식단 기록 수정 |
| RC-0115 | `DELETE /diet/records/{id}` | 식단 기록 삭제 |
| RC-0116 | `GET /diet/records` | 날짜별(`date=`) 또는 월별(`year=&month=`) 기록 + 합계 조회 |
| RC-0117 | `DELETE /diet/upload/{id}` | 확정 전(draft) 업로드 취소. 이미 확정된 기록(`analysis_status=COMPLETED`)은 409 — `/diet/records/{id}`로 삭제해야 합니다. |

**인증 방식 — 당시엔 신규 9개만 헤더 방식이었지만, 지금은 전 서비스가 헤더도 받습니다.** 아래 7번 항목에서 기존 엔드포인트에도 `Authorization: Bearer`를 병행 지원하도록 확장했습니다 — 최신 상태는 7번을 참고해주세요.

**RC-0113 입력에 없는 값 처리**: 명세 INPUT에 `serving`(단위 없는 숫자)만 있고 단위/탄수화물 필드가 없어서, `serving_unit`은 "인분" 고정값, `carbohydrate`는 0으로 저장합니다(실제 DB 컬럼이 NOT NULL이라 값을 비워둘 수 없음). 나중에 명세에 단위/탄수화물 필드가 추가되면 그때 반영하면 됩니다.

찜 테이블(`product.product_favorites`, `recipe.recipe_favorites`)은 `service` 스키마가 아니라 각 서비스 전용 신규 스키마에 만들었습니다 — community-service의 `community.notice_likes`와 같은 패턴이라, 데이터팀 소유 테이블은 전혀 건드리지 않았습니다.

## 6. 별도 논의 필요 — "제로파인더 기능명세" 시트

명세서에 새로 추가된 시트로, SR-/PD-/SW-/RC-/ST-/EV- 코드 체계를 쓰는 완전히 별도의 기능 목록입니다 (OCR 스캔 파이프라인, 매장 연동, Kafka 이벤트/모니터링 인프라 등). 저희가 만든 8개 서비스 중 어디에도 아직 배정되지 않았고, 상당수(EV-02xx 등)는 백엔드보다 데이터/인프라팀 영역으로 보입니다. 어느 서비스가 담당할지 팀장님 확인 부탁드립니다.

## 7. 2026-07-19 — PRODUCTION_HANDOFF.md P0/P1 반영

프론트팀 휴가 기간에 넘겨받은 `PRODUCTION_HANDOFF.md` 요청 사항 중 아래를 반영했습니다.

**P0-1 (OAuth 로그인 응답 없음)** — 원인은 login-service의 Redis 클라이언트에 타임아웃이 아예 없던 것이었습니다. Redis 연결이 안 되는 상황(방화벽 등)에서 `create_state()` 호출이 끝없이 멈춰, 겉으로는 "응답 없음"으로만 보였습니다(정확히 8초 뒤 프론트 타임아웃으로만 감지됨). `socket_connect_timeout`/`socket_timeout`(각 3초)을 추가해서, 이제는 Redis가 안 되면 3초 안에 명확한 에러로 실패합니다 — 근본적인 Redis 접근성(방화벽/네트워크) 자체는 별도 확인이 필요합니다.

**P0-3 (사진 업로드 흐름)** — `POST /diet/upload`가 `mealType`/`eatenAt`을 선택값으로 받습니다(안 보내면 기존 동작 그대로). `GET /diet/ai-analyze`를 Claude Vision(`claude-opus-4-8`)으로 실제 구현했습니다 — 사진 속 음식 항목(이름/제공량/칼로리/당류/탄수화물)을 추정해서 저장합니다. 서버에 `ANTHROPIC_API_KEY`가 없으면 기존과 동일하게 `PREPARING`을 반환합니다(무비용 폴백, 키 안 넣으면 배포해도 동작 그대로). 업로드 취소 API(`DELETE /diet/upload/{id}`)는 지난번 RC-0117로 이미 구현돼 있었습니다.

**P0-4 (인증 전달 방식)** — 전 서비스(login/product/diet/ingredients/main/community)의 기존 엔드포인트가 `usr` 쿼리파라미터에 더해 `Authorization: Bearer {JWT_TOKEN}` 헤더도 받습니다(헤더가 있으면 헤더 우선, 없으면 기존 `usr` 사용 — 완전히 하위호환). 프론트는 원하는 시점에 헤더 방식으로 옮겨가면 되고, 지금 당장 아무것도 안 바꿔도 기존 호출은 그대로 동작합니다. admin-service는 원래부터 헤더 전용이라 변경 없습니다. recipe-service는 이번에 처음 인증을 다루는 신규 엔드포인트(찜)만 있어서 헤더 전용으로 남겨뒀습니다.

**P1-1 (`/search` 카드 필드)** — `brand`/`category`/`serving`/`sugar`/`calories`/`image`/`tags` 필드와 `total`/`page`/`pageSize`/`hasNext`를 추가했습니다. 태그는 페이지 전체를 한 번에 조회해서 상품마다 따로 조회하는 N+1을 피했습니다.

**P1-2 (`/recipes` 페이지네이션/필터)** — `source`(10000recipe/youtube) 필터, `sort=sugarReduction`, `page`/`pageSize`/`total`/`hasNext`를 추가했고 카드에 `sugar`/`calories`/`source`를 채웠습니다. **`category`/`time`(조리시간)은 명세엔 있지만 실제 `service.recipes` 테이블에 해당 컬럼이 없어서 아직 못 채웠습니다** — 데이터팀에 컬럼 추가 필요 여부 확인 부탁드립니다.

**P1-3 (캘린더 집계 N+1 제거)** — `GET /diet/records?year=&month=`가 날짜별로 묶어서 각 날짜의 합계(`sugar-total`/`calories-total`)와 항목 목록을 한 응답에 담습니다. 기존 `/diet/calender`(RC-0106)는 혹시 이미 붙어 있는 곳이 있을까봐 그대로 뒀습니다 — 새 캘린더 화면은 `/diet/records` 쪽을 쓰시면 됩니다.

**아직 안 건드린 것** — P0-2(식단 CRUD 계약)는 이미 RC-0113~0117로 반영돼 있었습니다. P1-4(찜 API)도 PR-0307/0308·RC-0111/0112로 이미 반영됨. P1-5(실제 랭킹/추천), P1-6(상품 데이터 품질), P2 전체(OCR, 챗봇 등)는 이번엔 손대지 않았습니다.

### 요약 (팀장님께 전달용)

1. `/b/` 프리픽스는 게이트웨이가 처리, 프론트가 `.env`로 관리 — 백엔드 조치 불필요.
2. API-Spec 시트 RC-01xx 번호가 기능명세 시트와 어긋나 있어 정리 필요 (실제 URL은 정상 동작).
3. OCR 검색(`/search/lens`), 식단 OCR 분석(`/diet/ocr-analyze`), 저당 레시피 목록(`/receipe/low-sugar`), 레시피 AI 비교 요약(`/receipe/{n}/ai`)은 명세엔 있지만 아직 미구현 — 우선순위 정해주시면 진행하겠습니다.
4. 찜(PR-0307/0308, RC-0111/0112) + 식단 기록 CRUD(RC-0113~0117) 9개 신규 엔드포인트 구현 완료.
5. **[신규]** PRODUCTION_HANDOFF.md P0 항목(OAuth Redis 타임아웃, 사진분석 실구현, 인증 헤더 병행 지원) + P1 항목(검색/레시피 카드 필드·페이지네이션, 캘린더 집계) 반영 완료. 7번 항목 참고. 레시피 `category`/`time` 컬럼은 데이터팀 확인 필요.
