## 1. 체크리스트 (Docker 공식 근거)

### 1-1. 신뢰할 수 있는 베이스 이미지 사용
- **공식 근거**: *"The first step towards achieving a secure image is to choose the right
  base image."* Docker Official Images, Verified Publisher, Docker-Sponsored Open Source
  이미지를 우선 사용 — 이미 보안 검토를 거친 이미지들.
- [x] **충족** — `python:3.12-slim`은 Docker Official Image.

### 1-2. 최소 베이스 이미지 사용
- **공식 근거**: *"A smaller base image not only offers portability and fast downloads, but
  also shrinks the size of your image and minimizes the number of vulnerabilities introduced
  through the dependencies."*
- [x] **충족** — 이미 `-slim` 변형 사용 중(전체 이미지 대신).

### 1-3. 베이스 이미지 버전을 digest로 고정 🔴
- **공식 근거**: *"By pinning your images to a digest, you're guaranteed to always use the
  same image version, even if a publisher replaces the tag."*
- [ ] **갭 발견** — 지금 `FROM python:3.12-slim`은 태그일 뿐이라, 퍼블리셔가 같은 태그에
  다른 이미지를 올리면(드물지만 발생 가능) 우리도 모르게 다른 이미지가 빌드됨. digest
  고정(`FROM python:3.12-slim@sha256:...`)이 재현성·보안 면에서 더 안전.

### 1-4. Non-root 사용자로 실행 🔴 (가장 중요한 갭)
- **공식 근거**: *"If a service can run without privileges, use USER to change to a non-root
  user."* 그리고 *"Avoid installing or using sudo as it has unpredictable TTY and
  signal-forwarding behavior."*
- [ ] **갭 발견** — 지금 Dockerfile에 `USER` 지시어가 전혀 없어서 **컨테이너가 root로 실행
  중**. 컨테이너 탈출(escape) 취약점이 터지면 root 권한까지 그대로 넘어가는 구조.

### 1-5. Multi-stage build로 최종 이미지 최소화
- **공식 근거**: *"Split your Dockerfile instructions into distinct stages to make sure that
  the resulting output only contains the files that are needed to run the application"*
  (공격 표면 축소 목적).
- [x] **현재는 해당 없음** — `ci_sandbox`는 컴파일/빌드 도구가 필요 없는 순수 Python 앱이라
  멀티스테이지가 지금 당장 이득이 크지 않음. **다만 프론트엔드(빌드 도구 필요) 이미지는
  반드시 멀티스테이지 적용 — 개발 착수 후 재검토 항목.**

### 1-6. 불필요한 파일/패키지 제외
- **공식 근거**: `.dockerignore` 사용 + *"Avoid installing extra or unnecessary packages just
  because they might be nice to have."*
- [x] **충족** — `.dockerignore`에 `.git`/`.github`/`.claude`/캐시/문서 파일 이미 제외 중.

### 1-7. 이미지 정기 재빌드 🟡
- **공식 근거**: *"To keep your images up-to-date and secure, rebuild your images regularly
  with updated dependencies"*(`--pull` 플래그로 베이스 이미지 최신화).
- [ ] **확인 필요** — CI/CD의 `docker build` 단계가 `--pull` 옵션을 쓰는지 확인 필요(베이스
  이미지가 로컬 캐시에 있으면 오래된 버전을 계속 쓸 수 있음).

### 1-8. 컨테이너당 단일 역할
- **공식 근거**: *"Each container should have only one concern."*
- [x] **충족** — `ci_sandbox` 컨테이너는 uvicorn 프로세스 하나만 실행.

## 2. 런타임 보안 체크리스트 (OWASP Docker Security Cheat Sheet 기반) — 신규

**1장이 "이미지를 어떻게 빌드하는가"였다면, 이 장은 "그 이미지를 어떻게 실행하는가"다.**
Dockerfile을 아무리 잘 써도, 실행할 때 root 권한/무제한 리소스로 띄우면 소용없다.

### 2-1. 불필요한 권한(Linux capability) 전부 제거
- **공식 근거(OWASP RULE #3)**: *"Drop all capabilities `--cap-drop all` and then add only
  required ones."* 그리고 명확히 경고: *"Do not run containers with the `--privileged`
  flag!!!"*
- [ ] 🔴 **갭** — 지금 `docker-compose.yml`에 `cap_drop` 설정이 전혀 없음(기본값 = 여러
  capability가 살아있는 상태).

### 2-2. 권한 상승 방지
- **공식 근거(OWASP RULE #4)**: *"security-opt=no-new-privileges"*로 setuid/setgid 바이너리를
  통한 권한 상승을 차단.
- [ ] 🔴 **갭** — 미설정.

### 2-3. 읽기 전용 파일시스템
- **공식 근거(OWASP RULE #8)**: *"Run containers with a read-only filesystem using
  `--read-only` flag"*, 임시 쓰기가 필요하면 `--tmpfs`와 함께 사용.
- [ ] 🟡 **갭** — 미설정. `ci_sandbox` 코드를 직접 확인한 결과 파일 쓰기 코드가 없어서 적용
  리스크 낮음(단, Python 바이트코드 캐시 쓰기는 read-only 상태에서 조용히 실패하고 넘어감 —
  앱 동작에는 지장 없음).

### 2-4. 리소스 제한
- **공식 근거(OWASP RULE #7)**: 메모리/CPU/재시작 횟수/파일디스크립터/프로세스 수를 제한해서
  서비스 거부(DoS) 공격 완화.
- [ ] 🟡 **갭** — 미설정. 지금은 컨테이너 하나가 VM 리소스를 무제한으로 쓸 수 있는 상태.

### 2-5. Rootless 모드 (Docker 데몬 자체)
- **공식 근거(OWASP RULE #11)**: *"the Docker daemon and containers are running as an
  unprivileged user"* — 컨테이너 탈출이 일어나도 호스트에서 권한 상승이 안 되게.
- [ ] ⚪ **여지** — 이건 Dockerfile/Compose 레벨이 아니라 harbor VM의 Docker 데몬 설정 자체를
  바꿔야 하는 인프라 변경이라 영향 범위가 큼. 지금 당장 적용은 보류, 별도 검토 필요.

### 2-6. 공급망 보안 확장
- **공식 근거(OWASP RULE #13)**: 이미지 출처 문서화, SBOM, 이미지 서명, 신뢰할 수 있는
  레지스트리 사용.
- [x] Trivy 스캔(RULE #9)은 이미 적용 중. SBOM/이미지 서명은 코드단 체크리스트 A03에 이미
  동일 항목으로 반영돼 있어 중복 기재 안 함(상호 참조).

## 3. `ci_sandbox` 1차 시범 적용 — 실제 Dockerfile 대조

**현재 Dockerfile** (`Dockerfile`, 저장소 루트):
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY ci_sandbox/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY ci_sandbox/app ./app
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**대조 결과**: 8개 체크리스트 중 5개 충족(1-1,1-2,1-6,1-8) + 1개 현재 해당없음(1-5) +
**2개 실제 갭**(1-3 digest 미고정, 1-4 non-root 미설정) + 1개 확인 필요(1-7).

**✅ 적용 완료** (2026-07-15, `Jhoon/secops/docker-image-hardening` 브랜치):
```dockerfile
FROM python:3.12-slim@sha256:c3d81d25b3154142b0b42eb1e61300024426268edeb5b5a26dd7ddf64d9daf28

# non-root 사용자로 실행 (Docker 공식 권장: USER로 전환)
RUN groupadd -r appuser && useradd -r -g appuser appuser

WORKDIR /app

COPY ci_sandbox/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY ci_sandbox/app ./app

RUN chown -R appuser:appuser /app
USER appuser

EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

digest는 Docker Hub Registry API(`registry-1.docker.io`)로 `python:3.12-slim`의 현재 manifest
list digest를 직접 조회해서 확보(`sha256:c3d81d25...`). CI/CD의 `docker build`에 `--pull`
플래그도 함께 추가해서 1-7 갭도 같이 닫음. 로컬에 Docker CLI가 없어 직접 빌드 테스트는
못 했고, **실제 검증은 self-hosted 러너의 CI 파이프라인에서 이뤄짐**(PR 열면 확인 가능).

## 4. `ci_sandbox` docker-compose.yml 대조 — 런타임 보안

**현재 설정** (`infra/ci-sandbox/docker-compose.yml`):
```yaml
services:
  ci-sandbox:
    image: harbor.hizero.local/dangdang/ci-sandbox:${IMAGE_TAG}
    container_name: ci-sandbox
    ports:
      - "8080:8000"
    restart: unless-stopped
```

**대조 결과**: 2장 6개 항목 중 런타임 보안 설정이 **전부 없음**(2-1,2-2,2-3,2-4 미설정).
Trivy 스캔(2-6)만 이미 적용 중.

**✅ 적용 완료** (capability 제거, 권한상승 방지, read-only, 리소스 제한 — Rootless 데몬
전환(2-5)만 인프라 영향이 커서 보류):
```yaml
services:
  ci-sandbox:
    image: harbor.hizero.local/dangdang/ci-sandbox:${IMAGE_TAG}
    container_name: ci-sandbox
    ports:
      - "8080:8000"
    restart: unless-stopped
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges:true
    read_only: true
    tmpfs:
      - /tmp
    environment:
      - PYTHONDONTWRITEBYTECODE=1
    mem_limit: 256m
    cpus: 0.5
```

## 5. 우선순위 정리

| 갭 | 우선순위 | 이유 |
|---|---|---|
| Non-root 사용자 미설정 | 🔴 필수 | 컨테이너 탈출 시 root 권한 직결, 수정 난이도 낮음 |
| Capability 미제거(`cap_drop`) | 🔴 필수 | OWASP 공식 경고 사항, 수정 난이도 낮음 |
| `no-new-privileges` 미설정 | 🔴 필수 | 권한 상승 방지, 수정 난이도 낮음 |
| 베이스 이미지 digest 미고정 | 🟡 권장 | 실제 발생 확률은 낮으나 공식 권장 사항 |
| `--pull` 옵션 확인 | 🟡 권장 | CI/CD 워크플로 확인만 하면 됨 |
| Read-only 파일시스템 미설정 | 🟡 권장 | 코드상 리스크 낮으나 검증은 배포 후 필요 |
| 리소스 제한 미설정 | 🟡 권장 | DoS 완화, 실제 필요 리소스량은 운영하며 조정 필요 |
| Rootless Docker 데몬 | ⚪ 여지 | harbor VM 전체 영향, 별도 검토 필요 |
| 프론트엔드 이미지 멀티스테이지 | ⚪ 여지 | 아직 프론트 이미지 자체가 없음 — 개발 착수 후 적용 |

## 6. 이 문서로 완성되지 않는 부분

1. **프론트/백엔드 실제 이미지 구조를 모름** — 빌드 도구(webpack, 컴파일러 등)가 필요한지,
   필요하다면 멀티스테이지 설계를 어떻게 할지는 실제 이미지가 나와야 정할 수 있음.
2. **베이스 이미지 digest는 지금 고정해도 금방 stale해짐** — 보안 패치가 나올 때마다 갱신
   필요, 일회성 조치가 아니라 운영 프로세스로 만들어야 함(재빌드 주기와 연결).
3. **적정 리소스 제한값(mem_limit/cpus)은 추측치** — 실제 트래픽 패턴이 없어 256m/0.5 CPU는
   임의로 넉넉하게 잡은 값. 운영하며 실측 후 조정 필요.
4. **Rootless Docker 데몬 전환은 harbor VM 전체(Harbor 자체 포함)에 영향** — 이 이미지
   하나만의 문제가 아니라 인프라 결정이라 이 문서 범위 밖.
