# Claude Code 효과적 사용 지침

> 출처: Anthropic 공식 문서 "Claude Code 모범 사례"
> https://code.claude.com/docs/ko/best-practices
> 핵심 전제: **Claude의 context window는 빠르게 채워지고, 채워질수록 성능이 저하된다. context는 가장 중요한 관리 자원이다.**

---

## 1. 작업을 검증할 방법을 함께 줘라 (가장 영향력 큰 단일 행동)

Claude는 자기 작업을 검증할 수단(테스트, 스크린샷, 예상 출력)이 있을 때 훨씬 잘한다.
명확한 성공 기준이 없으면 "그럴듯하지만 실제로는 안 되는" 결과물을 만든다.

- **검증 기준 제공**: "이메일 검증 함수 만들어줘" (X) → "validateEmail 함수를 작성하고, 예제 테스트 케이스(a@b.com=true, invalid=false)로 검증한 뒤 테스트를 실행해줘" (O)
- **UI는 시각적으로 검증**: "스크린샷을 찍어 원본과 비교하고 차이를 나열한 뒤 수정해줘"
- **증상이 아닌 근본 원인 해결**: "이 오류로 빌드가 실패한다. 근본 원인을 해결하고 오류를 억제하지 말아줘"

---
## 2. 프롬프트에 구체적 컨텍스트를 줘라

지시가 정확할수록 수정이 적어진다. Claude는 의도를 추론하지만 마음을 읽지는 못한다.

- **작업 범위 지정**: 어떤 파일, 어떤 시나리오, 테스트 선호("모의 객체 피해줘")까지 명시
- **소스 지목**: "git 히스토리를 보고 이 API가 어떻게 변해왔는지 요약해줘"
- **기존 패턴 참조**: "X.php가 좋은 예다. 이 패턴을 따라 새 위젯을 만들어줘"
- **증상 설명**: 증상 + 의심 위치 + "수정됨"의 모습을 제공

> 풍부한 입력 방법: `@파일경로`로 파일 참조, 이미지 붙여넣기, `cat error.log | claude`로 파이프.

---

## 3. 효과적으로 소통

- **코드베이스 질문**: 선임 엔지니어에게 묻듯 직접 물어라. "로깅은 어떻게 동작해?", "이 줄은 왜 foo() 대신 bar()를 호출해?" → 온보딩에 탁월.
- **Claude가 나를 인터뷰하게 하기**: 큰 기능은 "AskUserQuestion 도구로 기술구현·UI/UX·엣지케이스·트레이드오프를 인터뷰한 뒤 SPEC.md에 사양을 작성해줘"로 시작. 사양 완성 후 **새 세션**에서 깨끗한 context로 구현.

---

## 4. 세션 관리 (대화는 지속·되돌리기 가능)

- **`Esc`**: 작업 중간 중지(context 보존, 방향 전환 가능)
- **`Esc + Esc` / `/rewind`**: 이전 대화·코드 상태로 복원
- **`/clear`**: 관련 없는 작업 사이에 context 완전 초기화
- **`/compact <지시>`**: 압축 시 초점 지정(예: `/compact API 변경에 집중`)
- **`/btw`**: 빠른 곁다리 질문(대화 기록에 안 들어가 context 증가 없음)
- **subagents로 조사 위임**: "subagents를 써서 X를 조사해줘" → 별도 context에서 탐색·요약 보고, 주 대화는 깨끗하게 유지
- **세션 재개**: `claude --continue`(최근 이어서) / `claude --resume`(목록에서 선택), `/rename`으로 세션 이름 지정

> **핵심 습관**: 한 문제를 두 번 이상 고쳤는데 또 틀리면, context가 실패한 접근으로 오염된 것. `/clear` 후 배운 점을 반영한 더 나은 프롬프트로 새로 시작하라. 누적 수정된 긴 세션보다 깨끗한 세션이 거의 항상 낫다.

---

## 5. 자동화·확장

- **비대화형 모드**: `claude -p "프롬프트"` (CI, pre-commit, 스크립트). `--output-format json`으로 결과 파싱.
- **병렬 세션**: Writer/Reviewer 패턴 — 세션 A가 구현, 세션 B(새 context)가 편향 없이 리뷰.
- **파일 전체 fan-out**: 파일 목록을 루프 돌며 `claude -p`로 분배. `--allowedTools`로 권한 범위 제한. 2~3개로 먼저 테스트 후 전체 실행.

---

## 6. 피해야 할 실패 패턴

| 패턴 | 증상 | 수정 |
| --- | --- | --- |
| 주방 싱크 세션 | 한 작업 → 무관한 질문 → 다시 첫 작업, context 오염 | 작업 사이 `/clear` |
| 반복 수정 | 고쳐도 또 틀리고 또 고침 | 2번 실패하면 `/clear` 후 더 나은 초기 프롬프트 |
| 과도한 CLAUDE.md | 너무 길어 규칙이 노이즈에 묻힘 | 무자비하게 정리, 이미 잘하는 건 삭제하거나 hook으로 |
| 신뢰-검증 간격 | 그럴듯하나 엣지케이스 미처리 | 항상 검증 제공, 검증 못 하면 배포 금지 |
| 무한 탐색 | 범위 없이 "조사"시켜 수백 파일 읽음 | 범위를 좁히거나 subagents로 격리 |

---

## 7. 직관 개발

이 패턴들은 시작점일 뿐 절대 규칙이 아니다.
복잡한 문제는 context를 쌓아야 할 때도, 탐색적 작업은 계획을 건너뛰어야 할 때도 있다.
잘 된 출력이 나왔을 때 내가 뭘 했는지(프롬프트 구조·제공한 context·모드) 주목하고, 막힐 때는 왜인지(context가 너무 시끄러웠나? 프롬프트가 모호했나? 작업이 너무 컸나?) 물어라. 시간이 지나면 가이드가 못 담는 직관이 생긴다.
